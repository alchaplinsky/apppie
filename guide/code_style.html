Coding Style

The purpose of this style guide is to define coding standart that should be used in frontend development to keep code consistency and create a baseline for collaboration. All code in any code-base should look like a single person typed it, no matter how many developers are contributing to it.

Structure

There are plenty of different methods for structuring a stylesheet. But chosen code format must ensure that code is easy to read and to make changes, easy to clearly comment and there is minimal chance of accidentally introducing errors.

  • Use soft-tabs (two space size), not spaces, to indent each property. Never mix spaces and tabs for indentation.
  • Add one blank line between blocks of css properties.
  • Include a single space between selector and opening curly brace. The closing brace should be flush left, using the same level of indentation as the opening selector.
  • Use one declaration per line in a declaration block.
  • Use one level of indentation for each declaration.
  • Include a single space after the colon of a declaration.
  • Use lowercase and shorthand hex values, e.g., #aaa.
  • Use double quotes consistently, e.g., content: "".
  • Quote attribute values in selectors input[type="text"].
  • Include a space after each comma in comma-separated property or function values.
  • Include a semi-colon at the end of the last declaration in a declaration block.

Incorrect 
.selector{ background:#fff; color: #333 }
Correct 
.selector {
  background: #fff;
  color: #333;
}

Selectors

Selectors are a powerful tool that allows us to be efficient. But this is also can be most frustrating part of cascading stylesheets. First of all you should use ONLY classes as selectors (except that case when you define default styles for links, headings, paragraphs, etc. in that case you use tags as selectors) and no ids.

  • Use lowercase and separate words with hyphens when naming selectors. Avoid camelcase and underscores.
  • Use human readable and semantic class names that describe what elements they style.
  • Avoid using over-qualified selectors, div.container can simply be stated as .container
  • Use js- prefix for classnames that are used by javasctipt. No styling should be applied to this classes!

Properties

  • All properties and values should be lowercase, except for font names and vendor-specific properties.
  • Use hex code for colors, or rgba() if opacity is needed. Avoid RGB format and uppercase, and shorten values when possible: #fff instead of #FFFFFF
  • Use shorthand (except when overriding styles) for background, border, font, list-style, margin, and padding values as much as possible.

Random properties ordering is chaos, it's better to order and group them, expecialy if you have a lot of them. A good choice is logical or grouped ordering, wherein properties are grouped by meaning and ordered specifically within those groups. The properties within groups are also strategically ordered to create transitions between sections, such as background directly before color. The baseline for ordering is:

  • Display
  • Positioning
  • Box model
  • Colors and Typography
  • Other

It may seem time consuming to think about properties order while writing css code. But once you get used to it, you'll see that everything has it's own place and you know exactly where you can find colors or positioning rules.

Values

There may be a lot of inconsistencies in values definitions. Following these simple guidelines will help you to avoid them:

  • Add single space before the value, after the colon
  • Do not pad parentheses with spaces
  • Use double quotes rather than single quotes, and only when needed, such as when a font name has a space.
  • 0 values should not have units unless necessary, such as with transition-duration.
  • Line height should also be unit-less, unless necessary to be defined as a specific pixel value.
  • Multiple comma-separated values for one property should be separated by either a space or a newline, including within rgba().

Media Queries

Media queries allow us to gracefully degrade the DOM for different screen sizes. If you are adding any, be sure to test above and below the break-point you are targeting.

  • Rule sets for media queries should be indented one level in.
  • It is generally advisable to keep media queries grouped by media at the bottom of the stylesheet.

Commenting

Since different code minifiers or uglifiers are more generally used, we can stop thinking about reducing stylsheets sizes by eliminating comments. Well commented code is extremely important. Take time to describe components, how they work, their limitations, and the way they are constructed. Don't leave others in the team guessing as to the purpose of uncommon or non-obvious code.

  • Place comments on a new line above their subject.
  • Keep line-length to a sensible maximum, e.g., 80 columns.
  • Make liberal use of comments to break CSS code into discrete sections.
  • Use "sentence case" comments and consistent text indentation.
